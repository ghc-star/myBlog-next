# 面经-AI智能博客

## 1. 项目简介（简历可用，1～2 句）

AI智能博客是一个基于 Next.js App Router、React 19、TypeScript 和 Tailwind CSS 搭建的个人技术博客，包含文章列表、分类、评论、点赞、访问统计和右侧 AI 助手等完整内容互动能力。项目的核心亮点是将 AI SDK、Embedding 和 Qdrant 向量库接入博客文章体系，通过 RAG 对文章内容进行语义检索，让用户可以用自然语言查询博客知识。

## 2. 简历 bullet（4～6 条）

- 基于 Next.js 16 App Router 设计博客页面与 API Route Handler，将文章首页、分类页、详情页等读数据链路放在服务端组件中，提升首屏稳定性与 SEO 友好度。
- 使用 React 19、TypeScript、Tailwind CSS 和 Framer Motion 实现文章列表、评论互动、侧边栏与 AI 抽屉等前端体验，兼顾响应式布局和交互状态管理。
- 接入 AI SDK 的 `streamText` 与 `useChat`，实现博客 AI 助手的流式问答、快捷问题、Markdown 渲染、加载态和错误态处理，降低用户等待感知。
- 设计文章 RAG 检索链路：离线将 MySQL 文章按 Markdown 标题和长度切分，调用 Embedding 服务生成向量，并写入 Qdrant 向量库用于语义召回。
- 封装 `getBlogStats`、`listArticles`、`readArticle`、`searchBlog` 等 AI tools，让模型根据用户意图选择确定性查询或向量检索，减少无依据编造。

## 3. 面试问题（按频率从高到低）

### 主题一：项目整体与前端架构

#### 1. 主问：请你整体介绍一下 AI智能博客这个项目，你主要解决了什么问题？

我的这个项目叫 AI智能博客，本质上不是一个只展示文章的静态博客，而是一个带内容互动和 AI 问答能力的技术博客。场景上，我希望用户不仅能通过列表、分类、文章详情去阅读内容，还能直接用自然语言询问“博客里有没有讲某个技术点”或者“推荐一篇相关文章”。我的任务主要是把 Next.js App Router 下的页面架构、前端交互和 AI 检索链路串起来。具体行动上，我用服务端组件承载文章列表、分类和详情这类偏数据读取的页面，用客户端组件承载 AI 抽屉、聊天输入、流式消息和主题交互；同时在服务端接入 AI SDK，把博客统计、文章列表、全文检索等能力封装成 tools。结果是项目从普通博客升级成了一个可以围绕文章内容对话的知识型博客，面试中我会把它定位成“前端主导、AI 增强”的工程项目。

#### 2. 追问：为什么这个项目适合用 Next.js App Router，而不是普通 React 单页应用？

当时我选择 Next.js App Router，主要是因为博客天然有 SEO、首屏速度和内容路由的要求。场景上，如果用普通 React SPA，文章列表和详情通常要等浏览器加载 JS 后再请求数据，这对首屏和搜索引擎都不是最优。我的任务是让内容页具备更好的服务端渲染能力，同时保留 AI 聊天、评论、主题切换这些客户端交互。行动上，我把首页 `src/app/page.tsx` 这类页面做成服务端组件，在服务端读取文章并处理分页；把 `AiChat`、`AiDrawer` 这类需要 `useState`、`useEffect` 和事件处理的部分标记为客户端组件。结果是项目能把“内容展示”和“强交互体验”拆开处理，既利用了 App Router 的服务端能力，也没有牺牲交互复杂度。

#### 3. 追问：你怎么划分服务端组件和客户端组件的边界？

我划分边界时遵循一个原则：凡是依赖数据库读取、环境变量、服务端 SDK 或首屏内容稳定性的部分尽量放服务端；凡是依赖浏览器事件、输入框状态、滚动、动画和本地状态的部分放客户端。具体场景里，首页文章列表只需要根据 `searchParams` 获取当前页并渲染文章，所以我放在 `src/app/page.tsx` 的服务端组件中；而 AI 聊天要处理输入、回车发送、loading 状态、错误提示和自动滚动，这些都必须运行在浏览器，所以 `src/components/ai/AiChat.tsx` 使用客户端组件。行动上，我避免把整个页面都写成 `"use client"`，而是把客户端能力收敛到叶子组件。结果是客户端包体和 hydration 成本更可控，也更容易向面试官解释项目的渲染分层。

### 主题二：AI 助手与流式对话

#### 4. 主问：你的 AI 助手从用户输入到返回答案，完整链路是什么？

这个链路可以分成前端输入、服务端编排、工具调用和流式返回四步。场景上，用户在右侧 AI 抽屉里输入问题，比如“博客里讲过 Next.js 迁移吗”。我的任务是让这个问题不是简单发给大模型，而是能结合博客内容回答。行动上，前端 `AiChat` 使用 `useChat` 管理消息和发送状态，提交后请求 `/api/chat`；服务端 `src/app/api/chat/route.ts` 解析消息，设置 system prompt，并通过 `streamText` 注册 `blogTools`；模型判断需要查博客内容时会调用 `searchBlog`，这个工具先把 query 转成 embedding，再去 Qdrant 搜索文章 chunk；拿到片段后模型组织中文回答，并通过 `createUIMessageStreamResponse` 流式返回。结果是用户能边等边看到回复，并且答案来源于博客内容而不是模型凭空发挥。

#### 5. 追问：为什么你要做流式响应，而不是等模型生成完一次性返回？

我做流式响应主要是为了解决 AI 交互里的等待感问题。场景上，用户问一个技术问题时，模型可能要先调用工具、再组织较长回答，如果服务端等全部生成完再返回，前端会长时间没有反馈，用户容易重复点击或者误以为卡住。我的任务是让用户能感知系统正在工作。行动上，我在服务端用 AI SDK 的 `streamText` 生成流式结果，并用 `createUIMessageStreamResponse` 转成前端可消费的 UI message stream；前端则通过 `useChat` 的 `status` 判断是否处于 `submitted` 或 `streaming`，在这个阶段禁用发送按钮并展示“思考中”。结果是交互更像实时聊天，长回答的体感延迟明显降低，也避免了重复提交导致的并发请求问题。

#### 6. 追问：AI 聊天前端你做了哪些体验细节？

我在 AI 聊天前端关注的不是只把接口调通，而是让它像一个可用的产品组件。场景上，用户可能第一次打开不知道问什么，也可能连续发送问题、查看长回答，甚至接口失败。我的任务是把这些状态都在组件里处理好。行动上，我在 `AiChat` 里加入了快捷问题入口，比如“博客有哪些文章”“推荐一篇文章”；发送时会 trim 输入并拦截空内容，在 loading 状态下禁止重复提交；消息区用 `ReactMarkdown` 和 `remarkGfm` 渲染模型返回的 Markdown；每次消息变化后滚动到底部；接口异常时展示错误提示。结果是 AI 助手不只是一个 textarea 加按钮，而是覆盖了空状态、加载态、错误态、长内容阅读和快捷引导的完整交互闭环。

### 主题三：RAG 检索增强生成

#### 7. 主问：你项目里的 RAG 是怎么设计的？

我项目里的 RAG 设计分为离线索引和在线检索两条链路。场景上，博客文章内容存在 MySQL 中，如果用户问具体技术问题，直接让大模型回答会有编造风险，也无法知道博客里真实写过什么。我的任务是把文章变成可检索的知识库。行动上，离线阶段我通过 `scripts/index-all-articles.ts` 从数据库拉取文章，把 Markdown 按 H2/H3 标题和长度切成 chunk，再调用 Embedding 接口生成向量，最后把向量和文章标题、URL、片段文本等 payload 写入 Qdrant。在线阶段，`searchBlog` 工具把用户 query 也转成 embedding，调用 `searchChunks` 找最相似的片段，再把这些片段交给模型生成回答。结果是 AI 助手能围绕博客文章做语义问答，回答更贴近项目内容，也更容易控制可信度。

#### 8. 追问：为什么文章要切 chunk，而不是整篇文章做 embedding？

我没有把整篇文章直接做 embedding，是因为整篇文章通常包含多个小主题，向量会变得很“平均”，用户问一个具体问题时很难精准召回。场景上，一篇文章可能同时讲背景、实现、踩坑和总结，如果整体向量化，某个局部技术点的语义会被稀释。我的任务是让检索粒度既足够精确，又不丢关键上下文。行动上，我在 `chunkMarkdown` 中优先按二级、三级标题切分，因为标题天然代表语义段落；如果某段太长，再按段落和长度继续拆分。结果是每个 chunk 通常能表达一个相对完整的小主题，Qdrant 返回的 topK 片段更容易直接支撑回答，也方便后续做召回准确率评估和 chunk 粒度调优。

#### 9. 追问：你怎么判断 RAG 检索效果好不好？

我不会只用“感觉回答还行”来判断 RAG 效果，而是会从召回和生成两个层面验证。场景上，AI 回答最终质量可能受 query embedding、chunk 切分、topK、模型总结能力共同影响，所以必须拆开看。我的任务是设计一个可复现的评估方式。行动上，我会准备 20 到 30 个来自真实文章的问题，记录 `searchBlog` 返回的标题、score 和片段文本，人工标注 topK 中是否命中正确文章、片段是否足以回答问题；生成层面再检查回答是否引用了召回片段，是否出现工具结果里没有的信息。结果上，目前项目还没有完整量化数据，所以我会在简历或面试中标注为（待测），但能清楚说明后续会用 TopK 命中率、片段可用率和编造率来评估。

### 主题四：向量库与 Embedding 工程实现

#### 10. 主问：Qdrant 在你的项目里承担什么角色？

Qdrant 在我的项目里承担的是语义索引和相似度检索的角色。场景上，MySQL 适合保存文章的结构化数据，比如标题、分类、发布时间和正文，但它不擅长回答“和这个问题语义相近的内容在哪里”。我的任务是给博客内容增加语义检索能力。行动上，我在 `src/lib/ai/vector.ts` 中封装了 QdrantClient，默认集合名是 `blog_articles_v1`，集合向量维度来自 Embedding 配置，距离函数使用 Cosine；写入时每个 point 包含 id、vector 和 payload，payload 保存 sourceType、sourceId、title、chunkIndex、text、url；查询时 `searchChunks` 返回 score 和 payload。结果是 AI tool 不需要关心底层向量库细节，只要传入 query vector，就能拿到可读的文章片段。

#### 11. 追问：你的 point id 为什么要做确定性生成？

我做确定性 point id，主要是为了让重复索引同一篇文章时可以覆盖旧数据，而不是不断产生脏数据。场景上，文章内容可能更新，索引脚本也可能多次执行，如果每次都随机生成 id，Qdrant 里会留下大量重复 chunk，检索结果可能命中新旧混杂的内容。我的任务是保证同一个 sourceType、sourceId 和 chunkIndex 对应同一个向量点。行动上，我在 `makePointId` 中用 `sourceType:sourceId:chunkIndex` 计算 sha256，再拼成 Qdrant 支持的 UUID 格式。结果是相同文章相同 chunk 重新 upsert 时能覆盖原 point，索引结果更稳定，也方便后续排查某个 chunk 的来源和版本问题。

#### 12. 追问：Embedding 维度、模型和环境变量你是怎么处理的？

Embedding 这部分我把模型配置做成环境变量，是为了避免代码和具体供应商强绑定。场景上，开发环境可能使用本地或第三方兼容 OpenAI 的 Embedding 服务，生产环境也可能更换模型或维度。我的任务是让代码只负责调用接口和处理返回向量，而不是写死所有配置。行动上，`src/lib/ai/embed.ts` 从 `EMBEDDING_BASE_URL`、`EMBEDDING_API_KEY`、`EMBEDDING_MODEL` 和 `EMBEDDING_DIM` 读取配置，默认模型是 `BAAI/bge-m3`，维度默认 1024；`embedTexts` 负责批量文本向量化，`embedQuery` 则封装单条 query 的场景。结果是索引脚本和在线检索可以复用同一套 Embedding 逻辑，也方便后续换模型时只调整环境变量和 Qdrant collection 配置。

### 主题五：AI Tools 与回答可信度

#### 13. 主问：为什么你把 AI 能力拆成多个 tool，而不是一个统一的搜索接口？

我拆成多个 tool，是因为用户问题的意图不同，最合适的数据来源也不同。场景上，用户问“博客有多少文章”时，答案应该来自确定性的文章统计；问“前端分类有哪些文章”时，应该查分类；问“某个技术点博客里怎么讲”时，才适合走语义检索。如果所有问题都进一个搜索接口，结果会不稳定，也浪费模型和向量检索成本。我的任务是让模型在不同意图下调用最准确的工具。行动上，我在 `src/lib/ai/tools.ts` 中定义了 `getBlogStats`、`listArticles`、`getArticlesByCategory`、`readArticle`、`searchBlog` 等工具，并用 Zod 描述入参；在 system prompt 里写清楚不同问题对应的调用规则。结果是模型能把确定性查询和 RAG 搜索区分开，回答可信度和可解释性更好。

#### 14. 追问：你怎么降低 AI 编造博客内容的风险？

我主要从提示词约束、工具结果约束和检索链路三方面降低编造风险。场景上，博客 AI 助手最怕的问题是用户问文章内容时，模型凭通用知识回答，听起来合理但和博客无关。我的任务是让模型尽量基于真实数据说话。行动上，我在 `src/app/api/chat/route.ts` 的 system prompt 中明确写了“不要编造工具里没有的内容”；对于博客整体、文章列表、分类这些问题使用确定性 tool；对于具体技术问题使用 `searchBlog`，返回标题、URL、score 和片段文本，让模型有上下文可依据。结果是回答会更倾向于引用博客已有内容。后续如果继续优化，我会要求回复中显式附上文章标题，并用人工测试集统计无依据回答比例。

#### 15. 追问：Zod 在你的 tools 里有什么作用？

Zod 在这里的作用是给模型调用工具时的参数提供结构化约束。场景上，AI tool calling 虽然由模型决定，但工具执行最终还是服务端代码，如果参数类型、范围不受控，就容易出现无效查询或者异常。我的任务是让每个工具的输入边界清晰。行动上，我在 `listArticles` 的 schema 中规定 `limit` 必须是 1 到 50 的整数，在 `searchBlog` 中规定 `topK` 是 1 到 10 的整数，在 `getArticlesByCategory` 中要求传入 categorySlug。结果是工具调用更稳定，服务端逻辑也更容易维护；面试中我会强调这不是为了“多写类型”，而是把大模型输出这种不确定输入收敛成可验证的服务端参数。

### 主题六：前端工程化与交互体验

#### 16. 主问：AI 抽屉这个交互为什么这么设计？

我把 AI 助手做成右侧抽屉，是为了让用户在阅读博客时可以随时提问，但不离开当前内容上下文。场景上，如果做成独立页面，用户需要在文章和 AI 页面之间来回跳转；如果做成底部常驻输入框，又会占用阅读空间。我的任务是在阅读体验和 AI 可达性之间做平衡。行动上，我用 `AiDrawer` 控制右侧固定抽屉，桌面端宽度约 420px，小屏下占满宽度；打开后延迟挂载遮罩，避免触发按钮的那次点击被遮罩立刻拦截关闭；关闭按钮和遮罩都能收起抽屉。结果是 AI 能力像一个辅助面板，而不是打断主流程的新页面，也更符合博客阅读时“边看边问”的使用场景。

#### 17. 追问：你在响应式和样式组织上怎么做？

我主要用 Tailwind CSS 和 CSS 变量来组织响应式布局和主题样式。场景上，博客既有桌面端的左右侧边栏，也要兼容移动端阅读，而且 AI 抽屉、文章卡片、评论区都需要统一的颜色体系。我的任务是让样式足够灵活，但不要把样式逻辑写得分散混乱。行动上，我在布局组件里通过 Tailwind 的响应式前缀控制间距和宽度，例如主布局给桌面端预留右侧栏空间；组件内部大量使用 `var(--text-main)`、`--card-bg`、`--theme-accent` 这类主题变量，保证切换主题时不需要逐个组件改颜色。结果是样式实现比较轻量，组件可读性也高，面试时可以从布局、主题变量和组件粒度三个角度展开。

#### 18. 追问：你如何处理聊天消息里的 Markdown 和代码展示？

我在聊天消息里使用 Markdown 渲染，是因为 AI 回答技术问题时经常包含列表、代码片段和强调内容。场景上，如果直接把模型文本用普通 `<p>` 展示，代码和列表可读性会比较差，尤其是博客技术问答场景。我的任务是让助手回答既能流式展示，又能保持技术内容的排版。行动上，我在 `AiChat` 中对 assistant 消息使用 `ReactMarkdown`，并接入 `remarkGfm` 支持 GitHub Flavored Markdown；同时给 prose、pre、code 设置了 Tailwind 样式，处理代码块背景、横向滚动和长单词换行。结果是用户看到的回答更接近技术文档阅读体验，而不是一段难读的纯文本，也更符合前端项目对细节体验的要求。

### 主题七：数据层、接口与安全边界

#### 19. 主问：博客除了 AI，还有哪些完整产品能力？

除了 AI 助手，项目还包含文章内容系统和互动系统。场景上，我不希望它只是一个展示 demo，而是接近真实博客产品，用户可以浏览文章、按分类筛选、查看归档、评论、点赞和统计访问量。我的任务是把这些功能拆到清晰的页面和 Route Handler 中。行动上，文章数据通过 `src/lib/article.ts` 和 MySQL 获取，首页、分类页、归档页分别承载不同阅读入口；评论、文章点赞、评论点赞、浏览量统计等能力通过 `src/app/api/comments/route.ts`、`src/app/api/article-likes/route.ts`、`src/app/api/comment-likes/route.ts`、`src/app/api/site-views/route.ts` 暴露。结果是 AI 助手不是孤立功能，而是建立在真实内容系统上的增强能力，面试时能展示我对前端产品链路的完整理解。

#### 20. 追问：你在接口层如何考虑输入校验和边界？

我在接口层会把用户输入和外部请求当成不可信数据处理。场景上，评论、点赞、聊天消息、天气城市名和 AI tool 参数都来自用户或模型，如果不做边界控制，容易导致异常、无效请求甚至安全问题。我的任务是保证 Route Handler 和 tool 执行时只接收符合预期的数据。行动上，在 AI tools 里我用 Zod 限制参数类型和范围；在聊天接口里会校验 messages 是否存在，空消息直接返回 400；对于天气城市名这类外部 API 参数，会用 `encodeURIComponent` 编码，避免拼接 URL 时出现问题。结果是服务端边界更加明确。后续我还会补充评论内容长度、登录态校验和敏感操作鉴权，让互动能力更接近生产标准。

#### 21. 追问：如果面试官问登录态或权限，你怎么回答？

我会如实说明当前聊天接口里登录校验代码存在但被注释掉，说明这是一个可以继续完善的点，而不是假装已经完全生产化。场景上，AI 问答接口会消耗模型和 Embedding 资源，如果完全开放，可能被滥用；评论和后台文章管理也需要明确权限边界。我的任务是区分“当前实现”和“生产化改造”。行动上，目前项目里有 `getCurrentUser` 和 GitHub OAuth 相关路由，说明已经具备接入登录态的基础；如果要上线，我会恢复 `/api/chat` 的登录校验，给管理端和写操作加权限判断，并对 AI 接口增加频率限制。结果是回答既诚实，又能体现我知道真实业务里鉴权、成本控制和滥用防护的重要性。

### 主题八：性能、测试与优化

#### 22. 主问：这个项目你会怎么做性能优化？

我会把性能优化拆成首屏性能、交互性能和 AI 响应性能三块。场景上，博客内容页最关注 LCP 和 SEO，AI 助手最关注首 token 时间和完整响应耗时，评论和点赞更关注接口响应稳定性。我的任务是先建立指标，再做针对性优化。行动上，首屏方面我会继续保持文章列表等内容在服务端获取，避免无意义的全客户端渲染；交互方面让 AI 抽屉和聊天状态只影响局部组件，避免全局重渲染；AI 方面会记录 `/api/chat` 的 TTFB、工具调用耗时和 Qdrant 检索耗时，必要时调整 topK、chunk 大小和模型选择。结果上，目前量化数据还处于（待测），但优化路径是清楚的：先测基线，再定位瓶颈，再做改动对比。

#### 23. 追问：你会如何测试 AI 助手是否稳定？

我会把 AI 助手测试分成前端交互测试、接口测试和 RAG 质量测试。场景上，AI 功能不稳定可能表现为消息发不出去、重复提交、流式显示断开、工具选错、召回无关内容或模型编造。我的任务是覆盖这些典型风险。行动上，前端我会手动测试空输入、Enter 发送、Shift+Enter 换行、长回答自动滚动、接口错误展示和移动端抽屉；接口层我会构造无 messages、空 messages、长问题等请求；RAG 层我会准备固定问题集，记录命中文章和回答是否有依据。结果是测试不只停留在“接口 200”，而是覆盖用户真实路径和 AI 质量路径。如果引入自动化，我会优先用 Playwright 覆盖聊天主流程。

#### 24. 追问：如果 RAG 回答不准，你会从哪些方向排查？

如果 RAG 回答不准，我会先判断问题出在召回还是生成，而不是直接换模型。场景上，回答不准可能是没搜到正确片段，也可能是搜到了但模型总结错了。我的任务是把链路拆开排查。行动上，第一步打印或记录 `searchBlog` 的 query、topK 结果、score、title 和 text，看正确文章是否进入候选；如果没进入，就检查 chunk 是否太大或太小、Embedding 模型是否适合中文、topK 是否太低、文章是否成功索引；如果正确片段已经召回，就检查 system prompt 是否明确要求基于工具结果回答，是否需要让回答附引用标题。结果是排查路径比较工程化，可以逐层定位，而不是把 AI 效果问题笼统归因给“大模型不稳定”。

#### 25. 主问：如果让你继续迭代这个项目，你优先做什么？

如果继续迭代，我会优先做三件事：AI 质量评估、权限与成本控制、以及前端可观测性。场景上，当前项目已经跑通了博客和 RAG 问答主链路，但要更接近生产项目，还需要证明效果、控制风险并能定位问题。我的任务是把 demo 能力升级成可持续维护的工程能力。行动上，第一，我会建立一套固定 RAG 测试集，统计 TopK 命中率、回答编造率和首 token 时间；第二，恢复聊天接口登录校验，加入频率限制和用量统计；第三，在前端和服务端记录关键错误与耗时，比如聊天请求失败、Qdrant 不可达、Embedding 失败。结果是项目不只是“功能能用”，而是能被验证、能被监控、能被持续优化，这也是我面向前端岗位想体现的工程意识。
